# Пайплайн разработки

## Полный цикл: «добавить новую фичу»

### Шаг 1. Постановка задачи

Юзер говорит: «Добавь систему кэширования для HTTP-запросов».

### Шаг 2. Архитектор загружает guardrails

GLM загружает skill `architecture-guardrails`, который содержит PRE-CODE и POST-CODE
чеклисты. Это гарантирует, что ни один принцип не будет нарушен.

### Шаг 3. GLM: план → уточнение → одобрение

Архитектор пишет план:

```markdown
## План

1. Создать CacheService с интерфейсом ICache
2. Реализовать MemoryCacheProvider
3. Интегрировать в HTTP-клиент проекта
4. Добавить юнит-тесты

### PRE-CODE чеклист
- [ ] Проверено: нет дублирования существующих сервисов
- [ ] Проверено: интерфейс отделён от реализации
- [ ] Проверено: обработка ошибок (cache miss, timeout)
- [ ] Проверено: потоко-безопасность
```

План показывается юзеру. Если нужны уточнения — `clarify`. После одобрения — шаг 4.

### Шаг 4. GLM: ТЗ для delegate_task

Архитектор формирует задачу для исполнителя:

```markdown
## Задача: CacheService

**Файлы**: src/services/cache/, tests/unit/cache/
**Конвенции**: PascalCase для классов, camelCase для методов, null -> Optional
**PRE-CODE чеклист**: (вставлен выше)

Создай:
1. ICache interface — 3 метода: get(key), set(key, value, ttl), invalidate(key)
2. MemoryCacheProvider — имплементация с TTL и потокобезопасностью
3. Интеграция в HTTP-клиент (HttpClient принимает ICache в конструкторе)
4. Юнит-тесты (xUnit/Jest/pytest — на выбор)
```

### Шаг 5. delegate_task #1: реализация

R1 (Исполнитель) получает задачу с чистым контекстом. Он:

1. Читает существующие файлы через `read_file` / `search_files`
2. Пишет новый код через `write_file` / `patch`
3. Собирает / запускает тесты
4. Возвращает self-report

**Self-report** (кратко):

```markdown
## Self-report

Создано:
- `src/services/cache/ICache.java` — интерфейс (3 метода)
- `src/services/cache/MemoryCacheProvider.java` — имплементация с ConcurrentHashMap
- `src/services/http/HttpClient.java` — изменён, принимает ICache

Тесты: 4/4 PASS
Проблем: не обнаружено
```

### Шаг 6. delegate_task #2: ревью

Второй R1 (Ревьюер) с независимым контекстом проверяет diff:

```
POST-CODE чеклист:
- [ ] Покрытие тестами > 70% новой логики
- [ ] Нет дублирования кода
- [ ] Все публичные методы документированы
- [ ] Нет хардкода (конфигурация вынесена)
- [ ] Обработка ошибок (cache miss не роняет приложение)
```

Результат: **✅ PASS** или **❌ FAIL** с комментариями.

### Шаг 7. Если FAIL → повтор

Если ревью не прошло, GLM отправляет исправленное ТЗ обратно R1.
Цикл повторяется до PASS.

### Шаг 8. Session-handoff

GLM загружает skill `session-handoff` и:

1. Обновляет `game_docs/CHANGELOG.md`
2. Обновляет `AGENTS.md` — добавляет информацию о новой фиче
3. Обновляет `game_docs/ROADMAP.md` — отмечает выполненное
4. Напоминает юзеру сделать `git commit`

### Шаг 9. Юзер: git commit

Человек смотрит diff, коммитит, пушит.

## Когда POST-CODE REVIEW не нужен

- Чисто документационные изменения (CHANGELOG, README)
- Правка навыков Hermes (skills/)
- Stub < 20 строк в новом файле
- Экспериментальный код, который будет выброшен

## Когда GLM делает ревью сам

**Никогда.** GLM не видит diff'ы. Если нужно ревью — второй delegate_task на R1.
Единственное исключение: GLM оценивает self-report на здравый смысл, но не проверяет код.

## Пример ТЗ для delegate_task (структура)

```markdown
## Задача: {название}

**Тип**: {новая фича / рефакторинг / багфикс}
**Пути**: {список файлов и папок}
**Конвенции**: {стиль кода, нейминг}

### Описание
{что нужно сделать, 2–5 предложений}

### PRE-CODE чеклист
- [ ] ...
- [ ] ...

### Критерии готовности
- [ ] Тесты проходят
- [ ] Нет регрессии
- [ ] Self-report заполнен
```
