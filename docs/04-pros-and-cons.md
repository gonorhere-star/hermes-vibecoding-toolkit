# Плюсы и ограничения

## Плюсы

### 1. Экономия токенов
GLM контекст остаётся чистым — никаких сырых файлов, diff'ов, логов сборки.
Токены тратятся только на осмысление и принятие решений. Сравните:
- **Один агент**: после 10 ходов ~50K+ токенов в контексте
- **Архитектор+Исполнитель**: после 10 ходов ~15K токенов в контексте

### 2. Архитектурная дисциплина
Guardrails не дают нарушать принципы. PRE-CODE чеклист проверяется ДО начала работы,
POST-CODE — ПОСЛЕ. Это автоматически ловит 80% типовых ошибок.

### 3. Изоляция контекста
`delegate_task` не загрязняет основной чат. Каждый запуск исполнителя — чистый лист.
Это значит, что модель не путается в предыдущих итерациях.

### 4. Параллельность
Можно запустить несколько R1 одновременно:
- Один пишет бэкенд
- Второй — тесты к нему
- Третий — документацию

GLM координирует, R1 работают.

### 5. Самодокументируемость
CHANGELOG, ROADMAP, ADR, AGENTS.md — всё это авто-генерируется на этапе session-handoff.
Проект всегда в актуальном состоянии, даже если вы забыли написать документацию.

## Ограничения

### 1. Не подходит для простых задач
Если задача < 20 строк кода — overhead от пайплайна (два delegate_task, чеклисты,
self-report) больше пользы. Просто сделайте руками.

### 2. delegate_task НЕ бесплатный
Каждый вызов `delegate_task` запускает R1, который тратит токены на загрузку
system prompt, инструментов и контекста задачи. На коротких задачах это может быть
дороже, чем сделать всё в одном окне.

### 3. Требует дисциплины
Один соблазн «дай-ка я (GLM) прочитаю файл сам» — и вся экономия исчезает.
Контекст начинает расти, дисциплина рушится. Нужно быть строгим к себе (или
настраивать автоматические блокировки инструментов через memory).

### 4. Не для команд > 3 человек
Система заточена под solo-разработку и малые команды. Нет координации между
разработчиками, нет разрешения конфликтов в параллельных ветках. Для больших
команд нужны традиционные code review и CI/CD.

### 5. Зависимость от качества R1
Исполнитель может галлюцинировать: придумать несуществующие API, неправильно
понять архитектуру. Self-report может быть неточным. Двойное ревью (R1 → R1)
снижает риск, но не устраняет полностью.

### 6. Сложнее дебажить
Когда что-то идёт не так, нужно разбираться: архитектор неправильно поставил
задачу? Исполнитель неправильно понял? Ревьюер пропустил баг? Это три точки
отказа вместо одной.
