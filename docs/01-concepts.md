# Концепция: Архитектор + Исполнитель

## Проблема

В типичном вайбкодинге один агент делает всё: читает файлы, пишет код, осмысляет архитектуру,
ревьюит. Это быстро приводит к **разрастанию контекста** — каждое прочитанное файловое дерево,
каждый diff — всё остаётся в истории. Через 3–4 итерации контекстное окно забито,
стоимость растёт, модель начинает ошибаться в простых вещах.

## Решение: разделение ролей

Мы делим разработку на две чёткие роли, которые общаются через `delegate_task`:

### GLM (Архитектор) — дорогая модель, низкий расход токенов

- **Планирование**: ставит задачу, пишет ТЗ, выбирает подход
- **Осмысление**: анализирует self-report от исполнителя, принимает решения
- **Session-handoff**: обновляет CHANGELOG, ROADMAP, ADR, напоминает о commit
- **НЕ трогает файлы** — не читает, не пишет, не собирает проект

GLM работает с минимальным контекстом: только план + self-report. Никаких сырых данных.

### DeepSeek R1 (Исполнитель) — дешёвая модель, высокий расход токенов

- **Реализация**: читает файлы, пишет код, правит существующий
- **Сборка**: компилирует, запускает тесты
- **Ревью**: проверяет свой diff по чеклисту
- **Self-report**: возвращает краткий отчёт архитектору

R1 может тратить миллионы токенов на чтение файлов и итерации — это нормально, он дёшев.

### delegate_task — изоляция контекста

Каждый вызов `delegate_task` создаёт **чистый контекст** для исполнителя.
Это значит:

- История основного чата не засоряется сырыми данными
- Можно запускать несколько исполнителей **параллельно**
- Исполнитель не видит предыдущие обсуждения архитектуры — только конкретную задачу

```
Юзер: "добавь систему X"
       ↓
  [GLM: Архитектор] ← дёшево в prompt, дорого в output
   - загрузить guardrails skill
   - написать ТЗ с PRE-CODE чеклистом
       ↓ delegate_task #1
  [R1: Исполнитель] ← дёшево, много токенов
   - читает файлы, пишет код, компилирует
   - возвращает self-report
       ↓ delegate_task #2
  [R1: Ревьюер] ← независимый контекст
   - проверяет diff по POST-CODE чеклисту
   - возвращает ✅ PASS / ❌ FAIL
       ↓
  [GLM: Session-handoff]
   - обновляет CHANGELOG / AGENTS / ROADMAP
   - напоминает git commit
```

## Почему не «один агент всё делает»

| Один агент | Архитектор + Исполнитель |
|---|---|
| Контекст растёт с каждым чихом | Контекст чистый, минимум истории |
| Токены горят на хранении сырых данных | Токены тратятся только на полезную работу |
| Модель «забывает» что было 3 шага назад | Каждая задача — свежий контекст |
| Нельзя распараллелить | Несколько R1 работают одновременно |
| Дорогой LLM читает все файлы проекта | Дешёвый R1 читает, дорогой — только отчёт |

## Когда это не нужно

- Простые правки (< 20 строк) — overhead больше пользы
- Быстрые эксперименты / прототипирование
- Когда вы один и контролируете контекст вручную
