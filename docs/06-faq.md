# FAQ

### «Зачем delegate_task, если можно просто читать файлы напрямую?»

Потому что каждый прочитанный файл остаётся в контексте GLM и раздувает его.
Через 5–10 прочитанных файлов контекст забит на 70%+ мусором, а не логикой.
`delegate_task` изолирует чтение в дешёвую модель — контекст архитектора остаётся
чистым, токены экономятся.

### «Какой моделью делать исполнителя?»

Рекомендуем **DeepSeek R1** (reasoner) — дёшево, точно, большое контекстное окно.
Альтернативы: GPT-4o-mini, Claude Haiku, Llama 3.1 70B (через API).
Главное — чтобы модель была (а) дешёвой и (б) справлялась с большим объёмом кода.

### «Что если R1 галлюцинирует?»

Двойная защита:
1. Второй R1 проверяет код через POST-CODE чеклист (независимый контекст)
2. GLM анализирует self-report на здравый смысл

Полностью исключить галлюцинации нельзя — но два прохода снижают риск до ~1–2%.

### «Можно ли использовать без UE5 / C++?»

**Да.** Этот тулкит не привязан к движку или языку. Работает с любым стеком:
Python, JavaScript/TypeScript, Go, Rust, C#, Java — что угодно. UE5 упоминается
только как один из примеров использования.

### «Сколько токенов экономит?»

~70–80% по сравнению со сценарием «один агент всё делает».
Точные цифры зависят от объёма проекта, но GLM (архитектор) тратит в 5–10 раз
меньше токенов при той же продуктивности.

### «Почему memory, а не config?»

Memory-записи попадают в system prompt **на каждом ходу**. Config — нет.
Блок-лист инструментов должен быть всегда перед глазами модели, иначе она
«забудет» о нём через пару ходов.

### «Что если я хочу сам читать файлы иногда?»

Можно, но это нарушает дисциплину. Как только GLM один раз прочитал файл сам —
контекст начал расти, и соблазн повторить усиливается. Если очень нужно —
выключите memory-блокировку, сделайте что нужно, и включите обратно. Но
старайтесь не злоупотреблять.

### «Можно ли заменить GLM на другую модель?»

Да. Любая достаточно умная модель подойдёт: Claude Sonnet/Opus, GPT-4o, Gemini.
Чем умнее — тем лучше архитектор. Но помните: ум = дорого, поэтому и нужно
ограничивать её работу.

### «У меня упал билд. Кто виноват — архитектор или исполнитель?»

Смотрите на self-report: если исполнитель написал код, который не компилируется —
виноват R1. Если ТЗ было некорректным (не указаны пути, не учтены зависимости) —
виноват GLM. Если чеклист пропустил баг — виновата конфигурация guardrails.
