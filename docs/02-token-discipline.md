# Токен-дисциплина

## Суть

**GLM (Архитектор) НЕ должен видеть содержимое файлов проекта.**
Всё, что он получает от исполнителя — это self-report: краткое описание что сделано, какие
файлы изменены, какие решения приняты. Ни сырых diff'ов, ни полного кода.

## Инструменты: стоп-лист для GLM

Инструменты, к которым GLM **НЕ ИМЕЕТ ДОСТУПА** (заблокированы через memory-запись):

| Инструмент | Причина блокировки |
|---|---|
| `read_file` | Чтение файлов = контекст растёт экспоненциально |
| `search_files` / `grep` | Поиск по проекту загружает много нерелевантных данных |
| `patch` / `write_file` | Архитектор не должен трогать код — это работа исполнителя |
| `terminal` (build) | Сборка/компиляция — задача исполнителя |
| `execute_code` | Запуск кода — только через delegate_task |
| `session_search` | История сессий не входит в задачу архитектора |
| `skill_manage` (правки) | Редактирование навыков — это модификация среды |

## Разрешённые инструменты для GLM

| Инструмент | Назначение |
|---|---|
| `clarify` | Уточнить требования у пользователя |
| `delegate_task` | Единственный канал связи с кодом |
| `todo` | Управление списком задач |
| `memory` (чтение/запись блок-листа) | Поддержание дисциплины |
| `cronjob` | Планирование повторяющихся задач |
| `skill_view` | Загрузка guardrails, чеклистов |
| `web_search` | Исследование, документация |

## Как enforce

1. **Memory-запись**: в профиле Hermes создаётся memory-запись с блок-листом инструментов.
   Она попадает в system prompt на каждом ходу.
2. **Coder-skill PRE-FLIGHT GATE**: навык исполнителя проверяет, что GLM не нарушил
   дисциплину перед тем, как начать работу.
3. **AGENTS.md**: контракт, который описывает роли и границы ответственности.
   Модель читает его при инициализации.

## Целевые показатели

| Метрика | Цель |
|---|---|
| GLM (архитектор) | < 5M токенов/день |
| Исполнитель (R1) | > 50M токенов/день |

Соотношение ~1:10. GLM тратит мало — только на осмысление. Исполнитель тратит много —
на чтение, запись, сборку.

## Почему это так важно

GLM в system prompt на каждом ходу = ~10 000–20 000 токенов.
Если GLM читает файлы:

- Один `read_file` = ещё ~500–2000 токенов в истории
- Один `search_files` = ещё сотни токенов
- После 10 ходов контекст уже забит на 50%+ файлами

Каждый такой токен — деньги, и каждый мешает модели сосредоточиться на архитектуре.

## Исключение

Тривиальный stub < 20 строк в **новом** файле. Если задача укладывается в пару строк
кода в новом файле — GLM может написать сам. Но как только это правка существующего
кода или больше 20 строк — только delegate_task.
