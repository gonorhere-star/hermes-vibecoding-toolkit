# ADR — Architectural Decision Records

> Узловые архитектурные решения. Не на каждый класс — только решения, меняющие структуру проекта.

---

## Когда писать ADR

- **Новый архитектурный паттерн** — вводим новый подход (например, Event Sourcing, Entity-Component)
- **Смена паттерна** — переходим с одного подхода на другой
- **Выбор альтернативы** — два+ равнозначных варианта, выбрали один
- **Breaking change публичного API** — изменение, ломающее обратную совместимость

## Когда НЕ писать ADR

- Новый класс в существующем паттерне — просто следуй установленному подходу
- Bug fix — почини и иди дальше
- Добавление свойства/метода — рутинная работа
- Косметический рефакторинг без изменения контрактов

---

## Шаблон ADR

Создавай файлы вида `NN-название.md` в этой папке.

```markdown
# ADR-NN: [название]

**Дата:** YYYY-MM-DD
**Статус:** Accepted

## Контекст

[Почему возник вопрос. Какая проблема решается. Какие ограничения.]

## Решение

[Что решили. Как именно реализовали / собираемся реализовать.]

## Альтернативы

1. **[Вариант A]** — [почему не подошёл]
2. **[Вариант B]** — [почему не подошёл]

## Последствия

- [+] [положительные последствия]
- [-] [отрицательные последствия / технический долг]
```

---

## Пример заполнения

```markdown
# ADR-01: Выбор ECS вместо наследования

**Дата:** 2026-07-20
**Статус:** Accepted

## Контекст
Иерархия классов сущностей разрослась до 7 уровней. Добавление новой механики требует правки в 3+ классах.

## Решение
Перейти на Entity-Component System. Сущность — пустой контейнер, логика — в компонентах.

## Альтернативы
1. **Глубокое наследование** — сохраняем текущий подход, но он уже показал свою негибкость
2. **Композиция через интерфейсы** — работает, но требует больше boilerplate чем ECS

## Последствия
- [+] Гибкость: новая механика = новый компонент
- [-] Нужно переписать существующие сущности
```
