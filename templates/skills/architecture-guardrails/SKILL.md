# Architecture Guardrails

## Секция 1: PRE-CODE чеклист

> Вставляй этот блок в `context` каждого `delegate_task` ТЗ перед началом реализации.
> **Важно:** Заполни `[ЗАПРЕТ N]` под свой проект перед использованием. Удали незаполненные плейсхолдеры.

```
⚠️ PRE-CODE АРХИТЕКТУРНЫЙ ЧЕКЛИСТ (обязательно к соблюдению):

❌ [ЗАПРЕТ 1: опиши свой] — пример: "Локальные константы вне общего файла конфигурации"
❌ [ЗАПРЕТ 2] — пример: "Глобальные переменные вместо dependency injection"
❌ [ЗАПРЕТ 3] — пример: "Логи через дефолтную категорию вместо именованной"
❌ [ЗАПРЕТ 4] — пример: "Прямой доступ к менеджерам вместо публичного API"
❌ [ЗАПРЕТ 5] — проект-специфичный запрет
❌ [ЗАПРЕТ 6] — проект-специфичный запрет
❌ [ЗАПРЕТ 7] — проект-специфичный запрет
❌ [ЗАПРЕТ 8] — проект-специфичный запрет

Каждый пункт — блокирующий. Нарушение = возврат на доработку.
```

## Секция 2: POST-CODE REVIEW чеклист

> Вставляй этот блок в `context` `delegate_task` для ревьюера после реализации.

```
✅ POST-CODE REVIEW:

- [ ] Код компилируется без ошибок?
- [ ] Нет нарушений PRE-CODE запретов?
- [ ] Нет новых логов через дефолтную категорию?
- [ ] `follow_decl` в .h где возможно?
- [ ] Нет дублирования кода (DRY)?
- [ ] Сборка зелёная?
- [ ] Нет хардкода там, где нужна конфигурация?
- [ ] Публичный API минимален и консистентен?
- [ ] Нет dead code / закомментированного кода?
- [ ] Обработка ошибок есть там, где нужна?
- [ ] Нет циклических зависимостей?
- [ ] Нейминг соответствует конвенциям проекта?

Формат ответа:
✅ PASS — если всё ок
❌ FAIL: [конкретный файл:строка] — [что нарушено]
```

## Секция 3: ADR-LITE

> Когда и как писать Architectural Decision Records.

### Триггеры для ADR:
- **Новый архитектурный паттерн** в проекте
- **Смена паттерна** (например, переезд с пулл- на пуш-модель)
- **Выбор альтернативы** между двумя равнозначными подходами
- **Breaking change** публичного API

### Когда НЕ писать ADR:
- Новый класс в существующем паттерне
- Bug fix
- Добавление свойства/метода к существующему API
- Рефакторинг без изменения контрактов

### Шаблон ADR (короткий):

```markdown
# ADR-NN: [название]

**Дата:** YYYY-MM-DD | **Статус:** Accepted

## Контекст
[почему возник вопрос, какая проблема решается]

## Решение
[что решили, как именно]

## Альтернативы
1. [Вариант A] — [почему не подошёл]
2. [Вариант B] — [почему не подошёл]

## Последствия
- [+] положительные
- [-] отрицательные / технический долг
```
